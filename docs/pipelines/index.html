Scheduled-task pipelines

Design surface for restructuring how background work runs in ChimpFlix. Three pipeline groups, mapped from current code with TODO badges on planned/pending changes.
Overview Media ingest + maintenance Watch state & housekeeping System tasks

Three pipeline groups

Tasks are grouped by what they touch. Media ingest and library maintenance are merged because the safety-net scheduled tasks call the same per-file job handlers as the on-add discovery pipeline — they are two entry points into one fan-out.

Media ingest + maintenance

File watcher, scanner, metadata agents, the per-file discovery fan-out (markers, loudness), the weekly safety-net sweeps, and the verify → purge cycle.

  • scan_library, refresh_metadata, refresh_logos, refresh_trending
  • detect_markers (incl. tacet integration)
  • analyze_loudness
  • fetch_subtitles, verify_libraries, purge_removed_files

Watch state & housekeeping

Hourly session reaper, hourly Trakt pull, daily audit-log + job-queue trim. These run on lighter cadence and don't require a maintenance window.

  • prune_sessions (hourly)
  • trakt_pull (hourly)
  • cleanup_audit_log, cleanup_jobs (daily)

System tasks

Database backup & verification, and the background optimized-version transcoder queue. Heavy I/O — all gated by the maintenance window.

  • backup_db (daily) → verify_backups (weekly)
  • optimize_versions (daily, capped concurrency)

Cross-pipeline connections

Edges that cross between the three groups. These are the "if I disable X does Y still work?" answers.

Edges between groups

  • Discovery → safety-net duality. The on-add discovery pipeline and the weekly safety-net scheduled tasks both call the same per-file handlers (detect_markers_file, analyze_loudness). Disable one entry point and the other still catches the file — provided idempotency keys (markers_detected_at, loudnorm_analyzed_at) keep retries cheap.
  • Scanner contention. The file watcher, the scheduled scan_library task, and admin-triggered "Scan now" all compete for a single per-library scan lock. Whichever wins runs; the others skip rather than queue. This is the same lock across all three entry points.
  • Verify → purge two-phase. verify_libraries (weekly) only soft-deletes (sets removed_at). purge_removed_files (daily) hard-deletes anything past the grace window (default 7 days). The split prevents losing watch state + markers on a temporary unmount.
  • Purge → cache eviction. purge_removed_files spawns async WebVTT subtitle cache eviction for the purged files. Crosses into the transcoder cache.
  • Trakt pull → watch state writes. trakt_pull writes item_watch_state.is_watched and resume positions for users with Trakt connected. Pull-only — Trakt push happens inline at playback end via play_state.rs, not on a schedule.
  • Optimize ← operator queue. optimize_versions consumes an operator-curated queue (admin UI). Respects transcoder_max_background_concurrent — it intentionally never competes with live playback transcodes.
  • Maintenance window is shared. Heavy tasks across all three groups (refresh_metadata, backup_db, analyze_loudness, optimize_versions, …) snap to the single configured maintenance window. Wrapping windows (e.g. 22:00 → 06:00) are supported.
  • Backoff layer. Any task failure adds exponential backoff (5 min → 6 hr cap) on top of its normal schedule. Shared mechanism, applies to every group.

Known gaps & pending work

Across all three pipelines

  • Pipeline gating by scheduled_tasks.enabled. The discovery pipeline fans out all four per-file kinds regardless of whether the corresponding scheduled task is enabled. Disabling detect_markers in the UI today still lets the on-add path enqueue marker jobs. (See project_pipeline_scheduled_task_gating memory.)
  • tacet integration. Intro/credit detection currently uses chimpflix_transcoder::detect_markers() — blackdetect + chapter parse + chromaprint fingerprint capture. The plan is to replace or augment with the standalone tacet crate at /mnt/github/tacet.
  • Manual watched/unwatched marking. Schema + scrobble path exist; UI + manual endpoint + Trakt push wiring still pending.
  • File-watcher toggle requires restart. scan_automatically is read once at startup. A runtime toggle would let operators stop the watcher without bouncing the server.
