Design notes for the restructure

  • File watcher is always-on by design. inotify is kernel-driven — there is no polling and the watcher consumes near-zero CPU when idle. It exists to react to Sonarr/Radarr/SABnzbd dropping files into library paths. scan_automatically is the only kill switch, read once at startup; we could promote it to a hot-reload setting if operators need to suspend ingest without restarting.
  • Automatic vs. Gated subgroups. Mirrors the original sketch. Automatic kinds (markers, preview sprite, logos) run on every new file/item with no per-feature toggle. Gated kinds (chapter thumbs, loudness, subtitles) are off by default; each has its own boolean in server_settings that admins flip in Library settings. Both entry points (on-add and safety-net sweep) check the gate.
  • tacet integration point. The detect_markers_file handler is the single hook. Today it calls chimpflix_transcoder::detect_markers() (blackdetect + chapter probe + chromaprint). Swapping it for tacet (from /mnt/github/tacet) is a one-handler change — the discovery pipeline + safety-net cron both pick up the new behaviour automatically because they share the handler.
  • Logos move from cron to pipeline. Currently a weekly sweep. Pulling it into the discovery pipeline (per-item, on first match) means new items get logos immediately instead of waiting up to a week. The weekly sweep stays as a safety net for items where the TMDB endpoint 404'd on first try.
  • Subtitles move from cron to pipeline. Same shape: per-item job enqueued when an item is first matched, gated by subtitle_fetch_enabled + OpenSubtitles credentials. Weekly safety-net sweep stays for items the on-add path missed (e.g. before the gate was on).
  • New pipeline kinds. Three new handlers join the pipeline:
    • detect_extras_item (Automatic, per-item) — scans for *-trailer.*, Extras/, Featurettes/, etc., and links them to the parent item. Fires on initial item match and again when an extras directory mtime changes. Plex-parity feature.
    • extract_embedded_subs (Gated, per-file) — pulls PGS/SRT/ASS streams out of MKV containers and writes .vtt sidecars. Pairs naturally with fetch_subtitles_item: extract first (free), fetch external only if no embedded track for the requested language. PGS OCR is the expensive case — keep concurrency at 1.
    • fetch_external_ratings (Gated, per-item) — OMDb / RT / MPAA pulls into a JSON column. Cheap network calls; rate-limited by external APIs so the 4-wide concurrency is conservative. Stale-after-30d sweep keeps figures fresh without thrashing.
  • Idempotency keys are load-bearing. chapter_thumbs_generated_at, loudnorm_analyzed_at, and per-language subtitle-presence checks are why the safety-net sweep stays cheap. Any new kind needs its own idempotency timestamp before it joins.
  • Per-file vs. per-item. Markers / sprites / chapter thumbs / loudness are per-file (each episode encode gets its own). Logos / subtitles / metadata are per-item (one set per movie or series). The pipeline needs both fan-out shapes; today the per-item enqueue path doesn't exist for the gated kinds yet.
  • Scan lock is per-library, not global. Two libraries can scan in parallel. The lock is only there to stop the watcher + cron + manual triggers from thrashing the same disk.
  • Maintenance window is the de-facto concurrency budget. Many tasks listed here all request the window — if it's narrow (default 02:00–09:00) and a backlog is large, they serialize. Could split into "io-heavy window" vs "network-heavy window" if it becomes contentious.
