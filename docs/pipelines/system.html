Design notes

  • Backup retention is unmanaged today. backup_db creates a new file each day; nothing prunes the directory. verify_backups walks the whole dir every week, so cost grows linearly. Worth deciding: keep N days, or keep last+weekly+monthly?
  • Verify is sufficient signal but not a real restore drill. PRAGMA integrity_check catches corruption but doesn't prove a restored DB starts up under the current schema. A periodic restore-into-test-instance would be the real check.
  • Optimize_versions concurrency cap is global, not per-GPU. If we add GPU device pinning per item, the cap probably becomes per-device rather than global so a multi-GPU host can run N encodes per card without operator math.
  • VACUUM INTO holds a read lock for the duration. Long enough on large DBs to delay writes. Maintenance window helps but doesn't eliminate the spike — could move to BACKUP API for incremental snapshots if it becomes noticeable.
  • Both system tasks share the same maintenance window as media ingest. If backup_db + optimize_versions + analyze_loudness + refresh_metadata all fire on the same night, they serialize through the scheduler. Splitting the window or assigning priorities is the lever if contention shows up.
