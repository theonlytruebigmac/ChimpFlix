Design notes

  • No maintenance-window gate on session/Trakt. Session reaper is a single indexed DELETE — fine to run hourly anytime. Trakt pull is network-bound and per-user, so the wall-clock cost is small even with many users.
  • Trakt push parity gap. trakt_pull can mark items watched without the user playing them locally (e.g. they watched on another device). When that happens, our manual mark-watched UI needs to push back to Trakt, otherwise the local mark gets overwritten by the next pull cycle. The pending watched/unwatched marking work has to wire the push side too.
  • Backoff applies here as well. If trakt_pull 401s for all users (e.g. global Trakt outage), the exponential-backoff layer prevents the hourly schedule from hammering. Worth confirming the 401 case classifies as a task-level failure, not a per-user one.
  • Audit log + job queue are coupled by retention. A failed job leaves both a job_queue row (with error) and an audit_log row. The two retention windows should probably be defined together rather than independently — a 90d audit row pointing at a job_queue row that vanished at 30d is confusing.
